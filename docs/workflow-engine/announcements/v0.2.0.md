# talos-workflow-engine 0.2.0

A Rust-native workflow engine for WASM-sandboxed module execution
with first-class AI/agent primitives and a signed job protocol for
dispatching to remote worker pools.

This is the first substantive release after 0.1 — closer to "ready
for early adopters" than a polish bump. Highlights below.

## What's new in 0.2.0

### A first-class pause primitive

`SystemNodeKind::Wait` is now an actual pause point. The engine
short-circuits the reactor and returns
`WorkflowContext { waiting: true, .. }`; resume by calling
`run_with_seed_with_transport` with the paused-node id mapped to
whatever value should stand in as its "output" (typically a webhook
payload, an approval verdict, an external trigger). The full
walkthrough is in
[`docs/checkpoint-lifecycle.md`](../checkpoint-lifecycle.md).

Multi-stage approvals (Wait → resume → Wait → resume) are covered
end-to-end by integration tests in
`tests/wait_pause_resume.rs`.

### Cancellation

Two new APIs:

* `engine.set_cancellation_token(Some(token))` — the
  non-`_cancellable` `run_*` methods consult it and short-circuit
  with `WorkflowEngineError::Cancelled` if the token fires.
  Propagates through `AdapterSet` so sub-workflow loops see the
  cancel too.
* `run_with_transport_cancellable(...)` / `run_with_seed_with_transport_cancellable(...)`
  — pass a token explicitly per-call (overrides the engine-level
  one). Useful for one-off runs.

In-flight worker abort is still out of scope (the engine has no
out-of-band channel to the worker pool); the dispatcher impl owns
that, via `DispatchJob::cancellation_token`.

### Configurable resource limits

Six new setters with documented defaults:

| Setter | Default | What it caps |
|---|---|---|
| `set_execution_timeout(Some(d))` | 5 min | Workflow wall clock |
| `set_max_workflow_nodes(n)` | 500 | `add_node` calls per engine |
| `set_max_node_output_bytes(n)` | 5 MiB | Per-node JSON output size |
| `set_max_fuel_per_node(n)` | 50M | Wasmtime fuel ceiling |
| `set_max_prefetch_successors(n)` | 8 | Speculative prefetch fan-out |
| `set_max_subflow_depth(n)` | 16 | Sub-workflow recursion depth |
| `set_agent_loop_max_history(n)` | 20 | Sliding-window context for agent loops |

Every limit propagates through `AdapterSet`, so sub-workflow loops
inherit the parent's overrides. Each is documented with "raise for
X / lower for Y" guidance.

### Sub-workflow recursion guard

A workflow that transitively references itself (the canonical
example: a `Judge` whose rubric calls itself) used to stack-overflow
the reactor. Now it terminates with
`WorkflowEngineError::SubflowRecursionLimit { depth, limit }` after
hitting the configured cap (default 16). Coverage in
`tests/subflow_recursion.rs`.

### `RateLimitStore` trait

Pluggable per-module rate-limit counter. Default is the in-memory
process-global one we shipped in 0.1; production deployments can
wire a Redis-backed (or other shared) impl via
`engine.set_rate_limit_store(...)` so caps hold across rolling
deploys and replicas.

The fail-mode is **fail-open** (a transport blip allows the
dispatch instead of blocking it) — a deliberate choice documented
on the trait.

### Fluent engine builder

```rust
let engine = ParallelWorkflowEngine::builder()
    .with_secrets_resolver(resolver)
    .with_user_id(user_id)
    .with_execution_timeout(Some(Duration::from_secs(60)))
    .with_max_workflow_nodes(1000)
    .with_cancellation_token(Some(shutdown_token))
    .build();
```

Same shape as `DispatchJobBuilder` (added in 0.2 too). The existing
`new()` + `set_*()` API still works; the builder is purely
ergonomic.

### Typed errors instead of `Execution(String)`

Five new variants promoted out of the catch-all:

* `Timeout { secs }` — wall-clock cap exceeded
* `ModuleFetcherMissing` — graph has module nodes but no fetcher wired
* `UserContextRequired` — graph has module nodes but no `user_id`
* `Cancelled` — caller cancelled via `CancellationToken`
* `SubflowRecursionLimit { depth, limit }` — recursion cap hit

`WorkflowEngineError` is `#[non_exhaustive]`, so adding more typed
variants in future patch releases doesn't break consumers who match
on what's currently typed.

### `InlineJudge` system node

For one-line scoring rubrics that don't need a separate
sub-workflow. Same `{score, passed, reasoning, feedback}` envelope
as `Judge`; evaluated synchronously through the configured
`ExpressionEvaluator`. Promote to a full `Judge` once the rubric
grows its own model call.

### Production-ready story

  `talos-workflow-engine-core` and `talos-workflow-engine`. Every

## Quickstart

```toml
[dependencies]
talos-workflow-engine-core = "0.2"
talos-workflow-engine      = "0.2"
talos-workflow-engine-nats = "0.2"   # if you use NATS
talos-workflow-job-protocol = "0.2"  # transitive via -nats; add directly for custom workers

[dev-dependencies]
talos-workflow-engine-test-utils = "0.2"
```

The fastest path into the codebase:

1. **Run the example.** `cargo run --example hello_workflow -p
   talos-workflow-engine` — a 3-node fan-out DAG end-to-end with
   in-memory adapters, no NATS, no wasm runtime.
2. **Read one cookbook.** Pick the integration guide nearest to
   what you'd actually build:
   [custom-dispatcher](../custom-dispatcher.md) /
   [checkpoint-lifecycle](../checkpoint-lifecycle.md) /
   [sub-workflow-composition](../sub-workflow-composition.md).
3. **Skim the production stack walkthrough.** When you're ready to
   wire it to real infra: [production-stack.md](../production-stack.md).

## Migrating from 0.1.x

The breaking changes are documented per-crate in each `CHANGELOG.md`.
Highlights:

* `DispatchJob::user_id` and `ChainDispatchRequest::user_id` are now
  `Option<Uuid>` (was `Uuid`). The migration is mechanical:
  `user_id: uid` → `user_id: Some(uid)`.
* `WorkerSharedKey` newtype wraps the HMAC signing key. Migrate
  `Some(Arc::new(bytes))` → `Some(WorkerSharedKey::new(bytes))`.
* `ParallelWorkflowEngine::run_with_transport` now returns
  `Result<_, WorkflowEngineError>` (was `Result<_, String>`). Most
  call sites just need to change the `?`/error binding type;
  `to_string()` still works for human-readable logs.
* The `MAX_WORKFLOW_NODES = 500` etc. constants are now per-engine
  setters with documented defaults — same values, different access
  pattern.

Full per-crate breaking changes are in:
- [`talos-workflow-engine-core/CHANGELOG.md`](../../talos-workflow-engine-core/CHANGELOG.md)
- [`talos-workflow-engine/CHANGELOG.md`](../../talos-workflow-engine/CHANGELOG.md)
- [`talos-workflow-engine-nats/CHANGELOG.md`](../../talos-workflow-engine-nats/CHANGELOG.md)
- [`talos-workflow-job-protocol/CHANGELOG.md`](../../talos-workflow-job-protocol/CHANGELOG.md)

## What's next (0.3+)

The roadmap section of the README has the full picture; the
short version:

* **0.3** — wire-format additions for worker-side cancellation
  (so `WorkflowEngineError::Cancelled` actually aborts in-flight
  WASM, not just the engine reactor). Coordinated controller +
  worker upgrade.
* **0.4** — public-API audit + crater-style regression sweep against
  reverse-dependency patterns.
* **1.0** — stable. Conditional on the above + at least one publicly-
  deployed production user willing to be a reference.

The trait surface in `talos-workflow-engine-core` is the part most
likely to survive 1.0 unchanged. The wire format in
`talos-workflow-job-protocol` is also stable for append-at-end field
additions during a coordinated rollout (snapshot-tested at the byte
level).
