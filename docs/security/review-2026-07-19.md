# Talos Security Review — 2026-07-19

**Scope:** Full-application security review of the Talos workspace (~105 crates,
controller + worker + frontend + Helm/k8s deploy). Conducted read-only across
seven domains (authentication/authorization, cryptography & secrets, SSRF &
outbound HTTP, multi-tenancy/RLS/SQL-injection, WASM sandbox isolation,
frontend, supply-chain/infrastructure), with each material finding verified
against source. One finding (P2, pipeline socket drift) was fixed as part of
this review; the rest are documented with concrete remediations.

**Headline:** The codebase is **exceptionally well-hardened**. The 57 structural
lint checks all pass clean, and the standard attack catalog for each domain is
already closed with tested, comment-documented mitigations. The findings below
are genuine gaps in that posture — most are defense-in-depth or
configuration-dependent rather than clean standalone exploits — but two
(P1/P2-High) are worth prioritising.

## P1 — Database SQL sandbox runs on the primary pool with no tenant scoping and no production guard  (High, config-dependent)

**Where:** `talos-rpc-subscribers/src/lib.rs:297` (`execute_guest_query`),
`:217` (`guest_role_for_query`), subscriber body `:1638`–`:2048`; wired with the
controller's primary pool at `controller/src/main.rs:3401`
(`spawn_database_rpc_subscriber(nats, db_pool.clone(), …)`).

**What:** A module compiled with the `database` capability world
(`database-node`) issues SQL over the signed `talos.database.query` RPC. The
controller subscriber re-validates the statement class (permits only
`SELECT/INSERT/UPDATE/DELETE/MERGE`, rejects DDL and deny-listed functions) and
then runs it via `execute_guest_query(&pool, …)` on the controller's **primary,
full-privilege** Postgres pool. The transaction sets **no** `app.current_org_id`
/ `app.current_user_id` GUC and applies **no** row/tenant filter. The only
privilege fence is `SET LOCAL ROLE "<TALOS_RPC_GUEST_ROLE>"`, which is:
- **opt-in** — unset by default, in which case guest SQL runs as the app user;
- **fail-open** — an invalid role identifier logs a warning and *disables* the
  wrap (`lib.rs:226`) rather than refusing to boot;
- **not enforced in production** — there is no boot guard, unlike the parallel
  fail-closed guards for RLS (`enforce_production_rls_posture`), env-KEK
  (check 45), and datastore TLS (check 44).

**Impact:** In the repo's documented in-cluster Postgres posture (controller
connects as a superuser, `TALOS_RLS_SET_ROLE` off, `TALOS_RPC_GUEST_ROLE`
unset), a tenant who can run a `database-node` module can
`SELECT`/`UPDATE`/`DELETE`/`INSERT` across **every tenant's rows** —
`workflow_executions`, `actor_memory`, `secrets`, `users`, integration/OAuth
tables, approval gates, etc. Cross-tenant **writes/deletes** are an integrity
breach regardless of encryption; cross-tenant **reads** expose all plaintext
metadata (workflow structure, schedules, key-paths, user emails, bcrypt hashes)
— envelope-encrypted columns (secret values, memory ciphertext) remain
encrypted, which bounds but does not remove the confidentiality impact.
Superusers bypass even `FORCE` RLS, so the P5 policies do not contain this in the
default posture.

The repo's own `docs/wasm-security-review-2026-05-22.md` (MEDIUM-2) already
identified "require `TALOS_RPC_GUEST_ROLE` at startup in production … the durable
fix" — it was never implemented.

**Remediation (recommended, in priority order):**
1. **Production boot guard (highest value).** Mirror
   `enforce_production_rls_posture`: in `is_production()`, refuse to boot when
   `TALOS_RPC_GUEST_ROLE` is unset **or** invalid, unless an explicit,
   SIEM-greppable `TALOS_ALLOW_UNSCOPED_DB_SANDBOX=1` acknowledgement is set.
2. **Fail closed on invalid role.** `guest_role_for_query` should refuse (not
   warn-and-disable) an invalid identifier in production.
3. **Scope the transaction.** Set `app.current_org_id`/`app.current_user_id`
   from the signed `actor_id → user → org` on the guest tx (route through
   `begin_org_scoped`) so RLS backstops it once P5 lands.
4. Add a structural lint sibling to check 44/45 asserting the guard exists.

## P2 — WASM pipeline execution over-grants raw `wasi:sockets`, bypassing `allowed_hosts`  (Medium-High) — FIXED

**Where:** `worker/src/runtime.rs` — pipeline step derivation (was
`!matches!(cap, Minimal | Unknown)`) vs the single-node
(`matches!(cap, Network | Database | Trusted)`) and sandbox paths.

**What:** `execute_pipeline_job` computed `allow_wasi_network` differently from
the single-node `execute_job` path. Single-node grants raw TCP/UDP sockets only
to the `Network`/`Database`/`Trusted` worlds (whose purpose *is* arbitrary
egress) and never to Tier-1. The pipeline path granted sockets to **every** world
except `Minimal`/`Unknown` — i.e. also to `Http`, `Secrets`, `Agent`,
`Filesystem`, `Messaging`, `Cache`, `Governance`.

**Impact:** Raw `wasi:sockets` bypass *both* the per-module `allowed_hosts`
allowlist and the host-fn tier gate (they are gated only by the private-IP SSRF
classifier). So an `http-node` (or a Python `secrets`/`agent` module, which can
resolve granted vault secrets) run as a **pipeline step** could open raw TCP to
any public IP and exfiltrate — a channel the *same module is denied in
single-node/sandbox execution*. This directly contradicted the documented
"layer-3" invariant at `select_tier` (`runtime.rs`). Tier-1 remained protected
(the `!Tier1` clause was present), so this is a Tier-2 `allowed_hosts`
containment/exfiltration bypass, not a Tier-1 privacy-ceiling break.

**Fix applied:** Introduced a single `socket_grant(cap, max_llm_tier)` helper
(`worker/src/runtime.rs`) as the sole source of truth for the socket-grant
predicate, and routed all three call sites (single-node, sandbox, pipeline)
through it. The pipeline path now denies sockets to `Http`/`Secrets`/`Agent`/etc.
exactly like the single-node path. Added `socket_grant_tests` covering the
per-world grant matrix and the Tier-1 denial. Recommend a follow-up structural
lint (sibling to check 38) asserting every `allow_wasi_network` derivation flows
from `socket_grant`, so the three sites cannot drift again.

## P3 — Pre-2FA (password-only) session can read the entire GraphQL query surface  (Medium)

**Where:** `controller/src/main.rs:8248` (injects `IsTwoFactorVerified(false)`
for a pre-2FA JWT but imposes no operation-level gate),
`talos-api/src/schema/mod.rs:114` (`require_scope` checks only `user_id`
presence — never `IsTwoFactorVerified`), vs `require_2fa` at `:135`
(mutation-side, correctly checks it). Independently confirmed by two reviewers.

**What:** After a password login on a TOTP-enrolled account, the server issues a
full session cookie whose JWT carries `is_2fa_verified=false`. Every GraphQL
**mutation** calls `require_2fa` and is blocked; the REST middleware
(`main.rs:8119`) rejects pre-2FA tokens outright with 403. But GraphQL **queries**
are gated only by `require_scope`, which passes for any session with a `user_id`.
So a pre-2FA session clears the entire read surface and WS subscriptions.

**Impact:** An attacker holding only the victim's password (the exact scenario
2FA defends against) can read the victim's data before completing TOTP —
including `actor_memories` (`talos-api/src/schema/actors/queries.rs:394`, which
**decrypts and returns** agent-memory content: emails/calendar/notes, up to
64 KiB/row), workflow definitions, execution history, and secret **metadata**
(key-paths, audit log). Tenant isolation still holds (victim's own data only)
and secret **plaintext** is never exposed (no `value` field), but this defeats
2FA for read-confidentiality. It is inconsistent with the codebase's own posture
everywhere else (REST + OAuth-callback both block pre-2FA).

**Remediation (pick one; #1 is smallest):**
1. Gate the GraphQL entry point: in `graphql_handler`, when the request is
   authenticated but `IsTwoFactorVerified(false)`, reject any operation other
   than the 2FA-completion / logout mutations (mirror the REST middleware).
2. Issue a restricted "2FA-pending" credential at login (accepted only by
   `verifyTwoFactor`/`logout`) instead of a full session cookie, so a pre-2FA
   cookie cannot authenticate arbitrary resolvers.

Note: whichever approach, verify the frontend's pre-2FA flow (fetching `me` /
enrolment state to render the TOTP prompt) is allowlisted so the fix doesn't
break the 2FA UI.

## P4 — Controller OCI catalog-sync uses a weaker Sigstore identity-regexp check than the worker  (Medium, defense-in-depth)

**Where:** `talos-registry/src/sync.rs:224` (controller — rejects only 8 literal
catch-all strings) vs `worker/src/module_fetcher.rs:334`
(`validate_sigstore_identity_regexp`, the canonical validator).

**What:** The worker's validator additionally rejects wildcard owner/repo
segments (`github.com/[^/]+/talos/…`), a missing `^` start-anchor, and a missing
`@` ref-anchor (the `workflow.yml-evil.yml` fork-spoof class). The controller's
ad-hoc matcher accepts all of those.

**Impact:** With an operator-set regexp like
`^https://github\.com/.*/talos/\.github/workflows/template-publish\.yml@`, the
controller catalog sync would run `cosign verify` with a wildcard owner and
accept a signature from *any* GitHub org's fork of that workflow, persisting
attacker-controlled template `name → oci_url` entries into the `modules`
discovery table. Bounded because under `Required` policy the *worker* refuses to
boot with such a regexp and never executes the redirected artifact — so this is
an Audit-window / discovery-surface DiD gap, not a clean standalone bypass. But
the controller is the layer CLAUDE.md advertises as "two-layer attestation," and
its layer is silently weaker.

**Remediation:** Promote `validate_sigstore_identity_regexp` to a small shared
crate (e.g. `talos-sigstore-policy`) and have `sync.rs::verify_oci_artifact_signature`
call it, so both processes enforce identical identity pinning.

## P5 — Postgres RLS is dormant by default and absent on the highest-value tenant tables  (Medium, defense-in-depth)

**Where:** `talos-db/src/lib.rs:24` (`TALOS_RLS_SET_ROLE`, default OFF); RLS
policies present on only ~12 tables (`actors`, `workflows`, `secrets`,
`workflow_executions`, `scratch_sessions`, `user_module_pins`, six `ml_*`).

**What:** Row-level security — the backstop that would contain P1 and P3 and any
future app-layer scoping miss — is off by default (when the app connects as
superuser, all policies are a no-op anyway), and has **no policy** on many
tenant-data tables that carry an `org_id`: `actor_memory`, `modules`,
`module_executions`, `integration_state`, `integration_credentials`,
`webhook_triggers`, `workflow_schedules`, `workflow_approval_gates`,
`workflow_suspensions`, `secret_audit_log`, and the OAuth-token tables
(`slack_integrations` / `gmail_integrations` / `google_calendar_integrations` /
`atlassian_integrations`).

**Impact:** No active leak on its own — app-layer scoping (`user_id`-scoped
reads/writes, signed `actor_id` for RPC) is currently intact and was verified.
But these uncovered tables are exactly what P1's sandbox reads, and there is zero
backstop if any single app-layer predicate is ever dropped.

**Remediation:** (1) Ship `FORCE` RLS policies for the remaining tenant tables,
prioritising `actor_memory`, `integration_credentials` + `*_integrations`,
`modules`, `module_executions`, `webhook_triggers`. (2) Move toward
`TALOS_RLS_SET_ROLE` on-by-default in production. (3) Route the P1 sandbox tx
through a scoped tx so these policies apply to it.

## Low-severity / hardening items

- **L1 — Cross-replica RPC replay fails open on shared-store outage.**
  `talos-replay-guard`: on `ReplayOutcome::Unavailable` (Redis blip) the default
  admits the request (`fail_closed_from_env()` defaults false). The Redis guard
  *is* registered in prod (`main.rs:504`), and per-replica HMAC+freshness+nonce
  still hold, so this is a documented availability tradeoff — but multi-replica
  high-assurance deploys should set `TALOS_REPLAY_FAIL_CLOSED=1` (consider making
  it the prod default for the write subjects).
- **L2 — Audit rotation keys skip the 256-bit entropy floor.**
  `talos-audit-event/src/lib.rs:287`: `TALOS_AUDIT_SIGNING_KEY_PREVIOUS` entries
  are accepted for verification without the `effective_key_entropy_bytes` check
  the primary key gets. A weak legacy key left in `..._PREVIOUS` weakens
  audit-forgery resistance during the rotation window. Apply the same floor to
  each previous key.
- **L3 — Login lockout branch is a timing oracle.** `talos-auth/src/lib.rs:774`:
  the locked-account path returns before any bcrypt verify (~100 ms faster),
  letting an attacker who burned the lockout threshold confirm account existence.
  Run the same dummy bcrypt as the user-not-found path.
- **L4 — API-key bcrypt truncates at 72 bytes.** `talos-api-keys`: the final 9
  chars of the 81-char key are never verified. ~220 bits still verified (no
  practical weakness today); pre-hash (SHA-256 → bcrypt) or assert the length
  invariant to avoid a footgun if the format grows.
- **L5 — Worker NATS connection lacks the controller's plaintext-scheme prod
  gate.** `worker/src/main.rs:1855` requires NATS *auth* in prod but no `tls://`
  scheme check; TLS is enforced only if `NATS_CA_FILE` is set. The shipped chart
  sets it, but an operator pointing the worker at plaintext `nats://` would
  transmit credentials in cleartext where the controller fails closed. Add a
  `// tls-prod-gate-nats` scheme check to the worker and widen lint check 44 to
  scan `worker/src`.
- **L6 — Frontend Dockerfile base images are unpinned.**
  `frontend/Dockerfile:2,19` use `node:20-alpine` / `nginx:alpine` by floating
  tag while all backend images pin by `@sha256:`. Pin both to digests.
- **L7 — Credential-bearing HTTP clients outside lint-check-49 scope.**
  `talos-github/src/client.rs:59` and `talos-google-cloud` build raw reqwest
  clients (currently correctly hardened) but aren't covered by check 49; a future
  edit dropping `.redirect(none())` on a token-bearing client wouldn't be caught.
  Add these crates to check 49's grep scope (or route through
  `talos_http_utils::trusted_client`).
- **L8 — External DLP provider response read is uncapped.**
  `talos-dlp-provider/src/lib.rs:654`: `reqwest::blocking … .json::<Value>()` has
  no body cap (operator-configured endpoint, so low risk of OOM). Add a
  `Content-Length` guard or a bounded blocking read.
- **L9 — `ControllerSsrfResolver` doesn't reset the resolved port to 0.**
  `talos-http-utils/src/outbound.rs:55`: returns resolved addrs with port 443
  verbatim; the worker's sibling resolver sets port 0 so hyper-util substitutes
  the scheme port. Latent *functional* bug (a controller webhook on a non-443
  HTTPS port would connect to 443), not an SSRF bypass. Mirror the worker.
- **L10 — Misc consistency:** CSRF rotation grace cache drops the cookie↔header
  binding for a 15 s window (`talos-csrf`); the custom
  `security/seccomp-{worker,controller}.json` profiles are not wired into the
  Helm chart (it uses `RuntimeDefault`) — dead artifacts; and `audit.toml` still
  lists `RUSTSEC-2024-0437` in its ignore block while `deny.toml` documents it as
  removed (doc drift, not a vuln).

## Domains verified clean (no high/critical)

- **Frontend:** No XSS sinks (`dangerouslySetInnerHTML`/`innerHTML`/`eval`
  absent), auth tokens in httpOnly cookies (not JS-readable storage), correct
  double-submit CSRF, strict `script-src 'self'` CSP, open-redirect allowlisting
  on all OAuth navigations, `noopener noreferrer` on external links.
- **Crypto & secrets:** AES-GCM nonces always `OsRng`; per-context HKDF subkeys
  collapse the birthday budget; constant-time MAC compares (`subtle`); AAD
  length-prefixed/separator-guarded against transposition; envelope-seal ECIES
  with contributory-check + forward secrecy; verify-once vs `verify_no_replay`
  split consistent; KEK all-zero/short/non-hex rejected; secret structs redact in
  `Debug`, held in `Zeroizing`; no plaintext credential fallback.
- **SSRF/outbound:** single-source IPv4/IPv6 classifier covering octal/hex/
  decimal/mapped/NAT64/6to4; connect-time DNS-rebinding resolvers on both
  controller and worker; every client `redirect(none())`; response bodies capped.
- **Auth/session (beyond P3):** bcrypt with cost floor/ceiling + dummy-hash
  constant-time user-not-found; 32-byte OsRng refresh tokens with rotation +
  reuse detection; algorithm-pinned JWT; HttpOnly+Secure+SameSite=Strict session
  cookies; TOTP AAD-bound + replay cache fail-closed + constant-time compare +
  advisory-lock backup codes; RFC-7239 right-to-left XFF walk; exact-match CORS.
- **WASM sandbox (beyond P2):** WASM proposals locked (threads/SIMD/GC off);
  `Component::new` panic-guarded (check 53); fuel + epoch + memory `ResourceLimiter`
  caps; tier HMAC-bound in the signing payload; AOT blobs HMAC-bound to
  version+cap-world+config; 5 tier-1 egress surfaces single-sourced.
- **Supply-chain/infra:** containers non-root + drop-ALL-caps + RO-rootfs +
  no-priv-escalation + seccomp `RuntimeDefault`; per-component default-deny
  NetworkPolicies with metadata/RFC1918 egress `except`; tightly-scoped RBAC;
  datastore TLS prod gates (check 44) + env-KEK guard (check 45); OCI digest +
  cosign identity/issuer pins; no `pull_request_target`; every advisory triaged
  with a documented non-reachability rationale.
